2. **Cursor-match tolerance is wide and already absorbing a 1-row offset.** `_cursor_matches` accepts ±1 row on both the reverse cell band and the live-cursor compare (terminal_screen.py:471-483). The sample's final frame shows `live_cursor.cursor_y=22` against `inferred_input_row=21`/reverse cell row 21 yet reports `cursor_matches_input_row: true` (last line of screen-metrics.jsonl). On an active frame this same ±1 band would mask a genuine one-row cursor drift / second-cursor regression. Consider tightening or documenting why ±1 is intended.

3. **Screen geometry is inferred from capture text, not the real pane.** `analyze_frame_files` calls `parse_ansi_screen(replay_data)` with no columns/rows (terminal_screen.py:362), so width = longest stripped line and height = line count (318-322). This only lines up with the real 100×30 pane because the TUI always draws full-width separators and tmux pads rows; a truncated/partial capture would shift column-sensitive findings (e.g., status at col 71) and the live-cursor row compare. The script knows the pane is `-x 100 -y 30` (script:147) — passing those explicitly would remove the fragility.

4. **`tmux capture-pane -J` joins wrapped lines** (script:61), which can merge a wrapped visual row into one logical line and desync screen-model row indices from the physical `#{cursor_y}` used for the live-cursor compare (script:63-66).
