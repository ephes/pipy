# Tool filter flags parity plan

Gap: Pi exposes `--tools/-t`, `--no-tools/-nt`, `--no-builtin-tools/-nbt`, and `--exclude-tools/-xt` in `packages/coding-agent/src/cli/args.ts`. Pipy currently lacks these product REPL flags, so every built-in plus extension tool is model-visible unless an extension dynamically narrows tools.

Scope for this slice: add the flags to `pipy repl` and top-level product dispatch, route them through the pipy-owned CLI/adapter/session boundary, and apply them to model-visible tool definitions and tool execution without changing extension APIs. Semantics should match Pi's `main.ts`/`core/sdk.ts` resolution: comma-separated `--tools` is the allow-list, comma-separated `--exclude-tools` is a deny-list applied after the allow/default set, `--no-tools` sets `noTools="all"`, and `--no-builtin-tools` sets `noTools="builtin"` only when `--no-tools` is absent. If `--tools` is present it controls the enabled names even when either no-tools flag is also present; `--exclude-tools` then removes matching names. Thus `--no-tools --tools read` enables `read`, and `--no-builtin-tools --tools read` also enables `read`, matching Pi's explicit allow-list precedence. Without `--tools`, `--no-tools` clears all model-visible tools including extension tools, while `--no-builtin-tools` removes only the canonical built-in registry names and leaves extension/custom tools available. Names are exact/case-sensitive after trimming empty comma segments, duplicates are harmless, and a name present in both `--tools` and `--exclude-tools` is excluded because the deny-list is applied after the allow-list. Unknown names should simply result in no matching registered tool, matching Pi's filter behavior; pipy has no existing tool-filter flag validation convention to preserve.

Reference paths: `/Users/jochen/src/pi-mono/packages/coding-agent/src/cli/args.ts` for flag parsing/help, `/Users/jochen/src/pi-mono/packages/coding-agent/src/main.ts` for CLI-to-session option precedence, `/Users/jochen/src/pi-mono/packages/coding-agent/src/core/sdk.ts` for `noTools`/allow-list/deny-list resolution, and `/Users/jochen/src/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts` for model-visible tools behavior.

Implementation shape:

1. Introduce a small `ToolSelectionPolicy` value object near the native tool-loop session boundary.
2. Add argparse flags to `pipy repl` (`--tools/-t`, `--no-tools/-nt`, `--no-builtin-tools/-nbt`, `--exclude-tools/-xt`) and ensure top-level `pipy ...` compatibility dispatch forwards them. Set the parser to `allow_abbrev=False`, configure these as explicit multi-character aliases (argparse does not bundle them), and test that `-nt`/`-nbt`/`-xt` parse as the intended aliases.
3. Thread the policy through `_tool_repl_adapter_for`, `PipyNativeToolReplAdapter`, and `NativeToolReplSession` so it applies equally to interactive REPL, `--print`, `--mode json`, and `--mode rpc`, all of which use the same adapter/session path. SDK/subagent-only surfaces are out of scope because pipy does not expose a separate public SDK tool policy yet.
4. Apply the policy after extension tools are merged into the run registry, before provider requests and actual tool execution. Keep a single canonical built-in-name snapshot from the pre-extension registry for `--no-builtin-tools`; that automatically tracks `production_tool_registry()` additions because the adapter/session already receives that registry.
5. Add focused tests for CLI parsing/help and session filtering, including precedence/combination cases (`--tools` plus `--exclude-tools` where overlap is denied, `--no-tools` plus `--tools`) and extension-tool survival under `--no-builtin-tools`.
6. Update parity docs (`docs/parity-plan.md`, `docs/pi-mono-gap-audit.md`, `docs/backlog.md`) and release notes.

Done when: focused tests and `just check` pass; docs show the tool filter flags as shipped; a different-family review returns CLEAN over the complete diff.
