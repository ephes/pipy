## What Changed

The `openai-completions` adapter now speaks Z.ai's thinking dialect. For a
reasoning-capable model whose thinking format resolves to `zai`, pipy emits a
single top-level boolean `enable_thinking` in the request body — `true` when a
thinking level is active, and an explicit `false` (a Pi-forced disable, not an
omission) when thinking is off or unset. It emits **no** `reasoning_effort` for
these models at all. This matches Pi's `thinkingFormat === "zai"` branch
(`openai-completions.ts:556-557`), so a glm-4.6-class model pointed at
`api.z.ai` now gets the exact request shape Pi sends instead of falling through
to the generic OpenAI `reasoning_effort` default.

Format detection grew a `zai` rung: a provider named `zai` or a base URL
containing `api.z.ai` resolves to the `zai` shape (an explicit
`compat.thinkingFormat` still wins over auto-detection). Critically, the rung is
inserted **before** `together` and `openrouter` in the detection chain, mirroring
Pi's `detectCompat` order (`isDeepSeek > isZai > isTogether > … > isOpenRouter`),
so a Z.ai row sitting on a Together- or OpenRouter-style base URL still resolves
to the `zai` shape rather than colliding with a later rung.

Unlike the DeepSeek and Together branches, the `zai` branch never consults
`supportsReasoningEffort`: setting `compat.supportsReasoningEffort=true` does
**not** add a `reasoning_effort` field, because the absence is structural to the
branch (Pi computes `enable_thinking = !!reasoningEffort` and emits nothing
else), not a side effect of Z.ai being on the effort-exclusion list. A new
conformance check (item `18j`) and unit tests pin both the on/off boolean and
the no-`reasoning_effort` invariant.

## Visualization

The crux is detection order plus the bare-boolean emission. Where each format
lands once detected:

```mermaid
flowchart TD
    A["thinking format detection<br/>(Pi detectCompat order)"] --> B{deepseek?}
    B -- yes --> BO["thinking:{type} + reasoning_effort*"]
    B -- no --> C{zai? (provider zai / api.z.ai)}
    C -- yes --> CO["enable_thinking: true|false<br/>NO reasoning_effort<br/>NO supportsReasoningEffort gate"]
    C -- no --> D{together?}
    D -- yes --> DO["reasoning:{enabled} + reasoning_effort*"]
    D -- no --> E{openrouter?}
    E -- yes --> EO["reasoning:{effort}"]
    E -- no --> F["openai default:<br/>top-level reasoning_effort"]
```

`*` = only when `supportsReasoningEffort` resolves true. The `zai` branch is the
only implemented rung with no such gate, and it sits ahead of `together` and
`openrouter` so collision cases resolve Pi-faithfully.

## Boundaries

- **Other `thinkingFormat` variants stay deferred.** `qwen`,
  `qwen-chat-template`, `ant-ling`, and `string-thinking` still fall through to
  the generic OpenAI `reasoning_effort` default — a documented deferral, not a
  regression. A full `detectCompat` port also remains a separate follow-on.
- **No clamping.** The off-state is gated on the raw off/unset thinking level,
  not merely on the level-mapper returning `None`. An unsupported clamped level
  emits neither `true` nor `false`, since pipy does not clamp — the same
  documented divergence already present on the DeepSeek/Together/OpenRouter
  paths.
- **No message-transform work.** This slice touches only the request reasoning
  shape; DeepSeek-style `requiresReasoningContentOnAssistantMessages` and any
  Z.ai response-side handling are out of scope.
