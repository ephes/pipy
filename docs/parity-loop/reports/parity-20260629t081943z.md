## What Changed

This slice made extension custom footers receive the same kind of footer-data object that Pi extensions expect. A footer factory registered with `ctx.ui.set_footer` now gets a read-only `FooterData` snapshot that can answer Pi-shaped calls for the current git branch, extension status map, and available provider count. Translated extensions can call either the Pythonic methods or the Pi camelCase methods (`getGitBranch()`, `getExtensionStatuses()`, `getAvailableProviderCount()`) without adapting their footer code.

The branch-change subscription surface also exists now: `onBranchChange(...)` accepts a callback and returns a disposer. In this slice it is intentionally a safe no-op because pipy still supplies a snapshot, not a live reactive footer-data provider.

## Boundaries

This did not ship live footer-data reactivity. Branch changes are not watched, registered `onBranchChange` callbacks are not invoked, and footer components are not automatically re-rendered when the branch changes. The slice also stayed within the footer-data provider surface; broader live component rendering, multi-widget message components, and richer tool-output expansion follow-ons remain deferred.
